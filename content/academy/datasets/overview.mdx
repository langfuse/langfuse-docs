# Datasets

## What this is

A dataset is a reusable set of test cases that you run your system against when you want to compare changes. It turns scattered production lessons into a structured artifact the rest of the loop can use repeatedly.

Each dataset item represents one situation your application should handle. The point is not to capture every possible input. The point is to capture the cases that matter enough to test on purpose.

## Why it matters

Without datasets, teams rely on memory and anecdotes. The same failure gets rediscovered in production. Experiments compare on ad hoc examples that change every time. Strong opinions win arguments because nobody is testing on the same cases twice.

A dataset makes learning durable. Once a case matters, you can keep testing it.

## When to use it

Build or expand a dataset when:

- A production issue keeps recurring.
- A failure mode is important enough that you do not want to rediscover it after every prompt or model change.
- You are about to compare a baseline and a candidate system and need shared inputs.
- You need coverage for risky edge cases, not just the most common traffic.

Datasets are the bridge between production monitoring and controlled experimentation.

## Core mental model

A dataset is not a random pile of examples. It is a deliberate sample of situations you care about, shaped so later experiments and evaluations can use it consistently.

### What a dataset item contains

Most dataset items have three fields:

- **Input**: the request, prompt, or situation you want to test
- **Expected output**: an optional reference answer or evaluation target
- **Metadata**: optional context that helps with filtering, slicing, or scoring

![Dataset Fields — When They Are Used](/images/academy/dataset-fields.png)

| Field | Purpose |
| --- | --- |
| Input | The situation or request you want to test |
| Metadata | Context that helps you segment or score the result |
| Expected output | A reference answer, label, or evaluation target when one is needed |

### Reference-based and reference-free datasets

Some dataset items need an expected output. Others do not.

- **Reference-based** items compare the system output against a known answer, label, or gold response.
- **Reference-free** items rely on criteria or evaluators that judge the output without a single ground-truth answer.

Both are valid. The right choice depends on the task you are testing and the evaluation method you plan to use later.

### Representativeness matters more than volume

The biggest dataset question is not “how many items do we have?” It is “what behavior does this set represent?” A tiny dataset built from recurring, high-value failure modes can teach you more than hundreds of convenience samples.

## What good looks like

A good dataset usually has these properties:

- Each item represents a real task or failure mode that matters.
- The dataset has a clear scope instead of mixing unrelated objectives.
- Metadata lets you slice results by scenario, customer segment, or risk category when that matters.
- Common cases and high-risk edge cases are both represented.
- The dataset is stable enough to compare results over time, but updated enough to reflect real production behavior.

## Common mistakes

- Treating a dataset like a random export of recent traces.
- Mixing different tasks or product surfaces into one set with no clear evaluation logic.
- Overfitting the dataset to the last incident instead of the broader failure pattern.
- Encoding too much evaluator logic into expected outputs until items become hard to interpret.
- Assuming more data is always better when the real problem is poor coverage.

## How it connects to the loop

Monitoring surfaces the cases that deserve to become durable tests. Experiments run different system versions against the dataset. Evaluation then uses those results to decide whether the change is actually better.

Datasets are what make learning reusable instead of reactive.

# Experiments

## What this is

An experiment is a controlled comparison between a baseline system and a candidate change. You run both on the same dataset, hold everything else as constant as possible, and compare the results.

The point is to isolate cause and effect. If the candidate looks better, you want to know whether it was because of the change you made, not because the inputs changed or the comparison was sloppy.

## Why it matters

Most teams can produce changes faster than they can judge them. Experiments create discipline. They stop prompt edits, model swaps, and architecture changes from being argued about purely on vibes.

Experiments are how you turn “this feels better” into “this is better on the cases we care about, at this cost, with this latency.”

## When to use it

Use experiments when:

- You want to compare a baseline against a prompt, model, retrieval, tool, or architecture change.
- A recurring problem deserves a measured fix, not a one-off patch.
- You need to understand a quality, cost, or latency tradeoff before shipping.
- You want evidence that a candidate change generalizes beyond one impressive example.

## Core mental model

An experiment has four parts:

1. **A baseline**: the current system or control condition.
2. **A dataset**: the shared set of inputs both conditions will run on.
3. **One deliberate change**: the prompt, model, context, tool access, or architecture element you want to test.
4. **A comparison**: the qualitative and quantitative evidence you will use later in evaluation.

### Change one variable at a time

The more variables you change at once, the less your result teaches you. A prompt rewrite plus a model swap plus a retrieval tweak is not one experiment. It is three guesses bundled together.

### Read outputs before you trust aggregates

Before you rely on scores, read outputs side by side. Open the traces for the same input under both conditions and ask:

- Which output is actually better?
- What kind of failure happened?
- Did the candidate improve the intended behavior or just move the problem elsewhere?
- Did cost or latency change in a way that matters?

Scores help you scale comparison, but they should not be the first time you look at the behavior.

## What good looks like

A good experiment usually has these properties:

- A clear hypothesis about what should improve.
- A strong baseline anchored in the current system, not an arbitrary comparison target.
- One primary variable changed at a time.
- The same dataset used across both conditions.
- Side-by-side output review before relying on aggregate scores.
- Cost and latency measured alongside quality.

## Common mistakes

- Comparing two fresh variants without anchoring to the current system.
- Changing multiple things at once and calling the result an experiment.
- Using a weak or unrepresentative dataset.
- Looking only at summary scores and never reading the outputs.
- Optimizing for one metric while ignoring cost, latency, or high-risk regressions.

## How it connects to the loop

Datasets provide the shared test cases experiments need. Experiments then generate the evidence that the [Evaluate](/academy/evaluate) step uses to decide whether a change is worth shipping.

After a change ships, monitoring tells you whether the improvement holds up in live traffic.

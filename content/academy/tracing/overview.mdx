# Tracing

## What this is

Tracing is the practice of recording what happened during one unit of work in your application. A useful trace captures the path a request took through the system and the evidence needed to understand why it behaved that way.

In LLM systems, that usually includes prompts, retrieved context, tool calls, inputs, outputs, model metadata, latency, token usage, and cost.

## Why it matters

Tracing is the foundation of the whole loop. You cannot monitor what you did not capture. You cannot build a trustworthy dataset from a half-recorded failure. You cannot compare experiment outputs if you do not understand what each run actually did.

Traditional logs or error reports are often not enough for LLM applications because technical success and user success are not the same thing. A model can return a valid response object and still fail the task.

## When to use it

Use tracing whenever you need to answer questions like:

- What prompt did the model actually receive?
- Which documents were retrieved for this answer?
- Why did the agent call that tool?
- Which step was slow or expensive?
- Was this failure caused by the model, the context, the tool chain, or application logic?

If your team is debating a production issue from memory, tracing is the missing layer.

## Core mental model

A trace is the record of one unit of work from the user's perspective. Inside it, spans record the individual steps that made that work happen.

### Trace, span, and session

- A **trace** is the end-to-end record of one task, request, or turn.
- A **span** is one step inside that trace, such as retrieval, a model call, or a tool invocation.
- A **session** groups related traces over a longer interaction, such as a full chat or workflow.

Choosing the right boundaries matters. A trace that is too broad becomes hard to reason about. A trace that is too narrow loses the context that later steps need.

### Traces are evidence, not just timing data

Good traces are not only trees of operations. They are evidence bundles. Each important span should answer:

- What was the input?
- What was the output?
- What metadata explains the behavior?
- How much time and money did this step consume?

### Useful span types

| Span type | Captures | Example |
| --- | --- | --- |
| Generation spans | A call to a language model | Full prompt or message history as input, the completion as output, plus metadata like model name, latency, and token counts |
| Retrieval spans | A step that fetches information from an external source | Query and the returned documents |
| Tool call spans | An invocation of a tool or function by an agent | Which tool was called, the arguments, and the return value |
| General/default spans | General processes | Highly dependent on the use case |

Span types are a reading aid. They make later monitoring and analysis faster because they let you filter for the steps that matter.

## What good looks like

A good tracing setup usually has these properties:

- One trace corresponds to one user-visible unit of work.
- Important steps are broken into spans with meaningful names.
- Model calls record prompt or message input, output, model name, token usage, latency, and cost.
- Retrieval and tool spans capture both the request and the returned result.
- Session identifiers exist when multiple traces belong to the same longer interaction.
- The trace is readable enough that another engineer can reconstruct what happened without opening application code.

## Common mistakes

- Recording only the top-level request and none of the internal steps.
- Capturing timing data but not the inputs, outputs, or retrieved context that explain behavior.
- Using inconsistent trace boundaries, so similar requests are not comparable.
- Recording every tiny function call, which creates noise without adding decision value.
- Tracing only failures instead of tracing normal behavior too, which makes baselines impossible.

## How it connects to the loop

Monitoring depends on traces to spot trends and surface suspicious cases. Datasets often start as production traces that represent recurring or high-risk scenarios. Experiments use traces to compare what the baseline and candidate system actually did.

If tracing is weak, every later step becomes more political and less empirical.

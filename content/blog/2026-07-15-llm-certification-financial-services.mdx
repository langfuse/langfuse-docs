One number turned the run red: 81.3%.

I had just finished a 150-item FinanceBench run with Claude Sonnet. The pipeline's default numerical-accuracy gate was 85%, so the model did not certify.

<Frame fullWidth>
  ![Certification results: Claude Sonnet on FinanceBench](/images/blog/2026-07-15-llm-certification-financial-services/benchmark-results.png)
</Frame>

Catching that failure on a dashboard before deployment, rather than in production after, is the argument of this post.

I have spent the past decade-plus helping financial services organizations build data systems. If you have ever shipped software next to a model risk management team, you know the rhythm: nothing reaches production until someone produces evidence, and producing that evidence by hand takes weeks.

That process was designed for credit models that changed twice a year. LLM systems change much faster. Prompts get promoted, models get swapped, tools and retrieval logic evolve, and production traffic drifts. Risk teams are not approving a raw model. They are approving the system built around it: a 10-K analysis assistant, a sentiment triage pipeline, or a client advisory drafting tool.

Here, **certification** means an internal, repeatable release gate and evidence package. It does not replace regulatory approval, independent validation, legal review, or accountable human sign-off.

The pipeline turns that evidence into a scripted process: it runs a model or agent against financial datasets, scores the output with domain-specific evaluators, and produces a PASS/FAIL verdict with a full audit trail in Langfuse. The code is available in the [langfuse-llm-certification-finance](https://github.com/doneyli/langfuse-llm-certification-finance) repository, pinned here to the [source state used for this post](https://github.com/doneyli/langfuse-llm-certification-finance/commit/86e14811c7647bbdf40f298d4ea0aca352e3e298).

That is the distinction I care about: evaluation scores alone do not certify an AI system. A certifiable system produces repeatable evidence across the model, prompt, agent trajectory, and human review. We will start with model evaluation, then go where the real risk conversations happen: [certifying the whole agent](#certifying-the-whole-agent), not just the model inside it.

## Golden datasets [#golden-datasets]

The pipeline uses two open-source financial benchmarks (a third, purpose-built adversarial set appears in the [agent section](#certifying-the-whole-agent)):

| Dataset                  | Source                                                                             | Items  | What it tests                                                                      |
| ------------------------ | ---------------------------------------------------------------------------------- | ------ | ---------------------------------------------------------------------------------- |
| **FinanceBench**         | [PatronusAI/financebench](https://huggingface.co/datasets/PatronusAI/financebench) | 150    | Financial Q&A from SEC filings: numerical extraction, reasoning, and justification |
| **Financial PhraseBank** | [ChanceFocus/en-fpb](https://huggingface.co/datasets/ChanceFocus/en-fpb)           | ~4,850 | Sentiment classification of financial news (positive, negative, neutral)           |

These are loaded into Langfuse as [datasets](/docs/evaluation/experiments/datasets), with each item containing an input (the question or text), an expected output (the correct answer or sentiment label), and metadata (source, question type, reasoning type):

```python
from langfuse import Langfuse
from datasets import load_dataset

langfuse = Langfuse()
ds = load_dataset("PatronusAI/financebench", split="train")

for item in ds:
    langfuse.create_dataset_item(
        dataset_name="certification/financebench-v1",
        input={
            "question": item["question"],
            "company": item.get("company", ""),
            "evidence": [ev.get("evidence_text", "") for ev in item.get("evidence", [])],
        },
        expected_output={
            "answer": item["answer"],
            "justification": item.get("justification", ""),
        },
        metadata={
            "question_type": item.get("question_type", ""),
            "question_reasoning": item.get("question_reasoning", ""),
            "source": "PatronusAI/financebench",
        },
    )
```

The `question_reasoning` field matters later: it is free ground truth for checking whether an agent took the right path, not just whether it produced the right answer.

One practical detail I lean on constantly: every dataset loads in two sizes. The setup script's `--sample` flag loads 10-item versions (`certification/financebench-sample`) that run end to end in a couple of minutes; that is what I use in live demos and as the fast inner loop while iterating on evaluators and prompts. The full set (`certification/financebench-v1`, all 150 items) is what a formal certification run gets scored against. The task, evaluators, and gates stay the same between the two sizes, which reduces configuration drift. The results can still change materially as coverage expands, so a sample run is a directional check, not certification evidence.

## Running the experiment [#running-the-experiment]

Each dataset item gets sent to the model under test. For FinanceBench items that include evidence excerpts from SEC filings, the prompt includes the source documents as context, simulating a RAG pipeline:

```python
def create_certification_task(model, endpoint, api_key):
    def task(*, item, **kwargs):
        inp = item.input if hasattr(item, "input") else item.get("input", {})
        question = inp.get("question", inp.get("text", ""))
        evidence = inp.get("evidence", [])

        if evidence and any(evidence):
            context = "\n\n".join(
                f"--- Source Document Excerpt {i} ---\n{ev}"
                for i, ev in enumerate(evidence, 1) if ev
            )
            prompt = (
                f"You are a financial analyst. Answer the question using ONLY the "
                f"provided source document excerpts. Be precise with numbers.\n\n"
                f"{context}\n\n--- Question ---\n{question}"
            )
        else:
            prompt = question

        return call_model(prompt, model, endpoint, api_key)

    return task
```

In the repository this template is not hardcoded: it is fetched from [Langfuse prompt management](/docs/prompt-management/get-started) by name, with the string above as a fallback. That detail matters in a regulated setting, because it means a compliance officer can edit, version, and promote the certification prompt in the Langfuse UI without a code deploy, and every run records exactly which prompt version it was certified against.

The task function is passed to [`dataset.run_experiment()`](/docs/evaluation/experiments/experiments-via-sdk), which handles concurrency, tracing, and evaluation in one call:

```python
from langfuse import get_client

langfuse = get_client()
dataset = langfuse.get_dataset("certification/financebench-v1")

result = dataset.run_experiment(
    name="financebench-v1",
    run_name="claude-sonnet-4-6-20260715",
    task=create_certification_task(model, endpoint, api_key),
    evaluators=[
        numerical_accuracy_evaluator,
        exact_match_evaluator,
        regulatory_compliance_evaluator,
        response_completeness_evaluator,
        groundedness_evaluator,
    ],
    run_evaluators=[
        average_score_evaluator("numerical_accuracy"),
        average_score_evaluator("groundedness"),
        certification_gate("numerical_accuracy", threshold=0.85),
    ],
    max_concurrency=5,
)
```

Every model call is traced in Langfuse and scored by the evaluators. Running the same dataset against multiple models produces a side-by-side comparison in the Langfuse UI.

## Evaluators [#evaluators]

The pipeline deliberately mixes two kinds of evaluators, because they catch different failure modes.

**Deterministic evaluators** (fast, cheap, reproducible):

1. **Numerical accuracy** extracts numbers from the output and compares them against the expected answer with a 5% tolerance, handling currency symbols, commas, percentages, and rounding differences.
2. **Exact match** checks whether the expected answer appears verbatim in the model output.
3. **Sentiment classification** compares the model's sentiment label against the ground truth from Financial PhraseBank.
4. **Regulatory compliance** scans model outputs for prohibited phrases like "guaranteed returns" or "risk-free investment."
5. **Response completeness** scores based on response length and structural formatting.

**An LLM-as-a-judge evaluator** for the dimension the deterministic checks cannot see: [groundedness](/docs/evaluation/evaluation-methods/llm-as-a-judge). A judge model (configurable via an environment variable) receives the output, the source evidence, and the question, and scores faithfulness and completeness against a financial auditor rubric, weighted 70/30.

The run that convinced me you need both came from the seeded 10-item FinanceBench sample: Claude Haiku's numerical accuracy was 60% while groundedness was 97%. Without the judge, the report says "60%, FAILED" and everyone concludes the model is unreliable. With it, you can tell the risk team something far more useful: the model faithfully uses the evidence it is given, and its failure is specifically numerical reasoning. Those are two very different remediation paths, and the distinction is exactly what a model risk assessment needs.

All evaluators follow the same Langfuse signature. Here is numerical accuracy, condensed from the repository:

```python
from langfuse import Evaluation

def numerical_accuracy_evaluator(*, output, expected_output, **kwargs):
    answer = expected_output.get("answer", "")
    expected_nums = _extract_numbers(answer)

    if not expected_nums:
        # Not a numerical question: fall back to string containment
        matched = answer.strip().lower()[:20] in str(output).strip().lower()
        return Evaluation(name="numerical_accuracy", value=1.0 if matched else 0.0)

    matched, detail = _numbers_match(expected_nums, _extract_numbers(str(output)))
    return Evaluation(
        name="numerical_accuracy",
        value=1.0 if matched else 0.0,
        comment=detail,
    )
```

## The pass/fail gate [#the-pass-fail-gate]

On top of the item-level evaluators, a run-level `certification_gate` evaluator produces a binary PASS/FAIL verdict. Run evaluators in the Langfuse SDK receive the full list of item results, so the gate averages the relevant score across the run and compares it to the threshold:

```python
def certification_gate(score_name, threshold=0.85):
    def evaluator(*, item_results, **kwargs):
        values = [
            ev.value
            for result in item_results
            for ev in result.evaluations
            if ev.name == score_name and ev.value is not None
        ]
        avg = sum(values) / len(values) if values else 0.0
        passed = avg >= threshold

        return Evaluation(
            name="certification_result",
            value=1.0 if passed else 0.0,
            comment=f"{'PASSED' if passed else 'FAILED'}: avg {score_name}={avg:.1%} (threshold={threshold:.0%})",
        )
    return evaluator
```

The results can be exported as review-ready evidence packages in Markdown, JSON, or CSV using the export script:

```bash

python run_certification.py --dataset certification/financebench-sample --model claude-sonnet-4-6
python export_results.py --dataset certification/financebench-sample --format markdown
```

## Certifying the whole agent, not just the model [#certifying-the-whole-agent]

Here is the part I care most about, because it is where the certification conversation actually happens. No risk committee approves "Claude Sonnet." They approve the 10-K analysis assistant built on it. The assistant plans, retrieves evidence, calls a calculator tool, and composes an answer. Certifying the model tells you nothing about whether that system is safe to deploy.

So the pipeline also certifies registered agents end to end. Each agent runs as a multi-step task per dataset item, with one nested span per step, so a reviewer can open any item in Langfuse and see exactly which tool the agent invoked and what it returned. And instead of a single accuracy threshold, each agent defines a **multi-dimensional gate**: every dimension must pass at once, or the certification fails.

| Agent              | Steps (spans)                                  | Gate (all must pass)                                                       |
| ------------------ | ---------------------------------------------- | -------------------------------------------------------------------------- |
| `10k-analyst`      | plan → retrieve-evidence → calculate → compose | accuracy ≥ 85%, groundedness ≥ 80%, compliance = 100%, tool use ≥ 90%      |
| `sentiment-triage` | classify → rationale → route                   | sentiment accuracy ≥ 85%, compliance = 100%, tool use = 100%               |
| `advisory-draft`   | analyze → draft → compliance-self-check        | groundedness ≥ 80%, compliance = 100%, completeness ≥ 70%, tool use = 100% |

```bash
python run_usecase_certification.py --list
python run_usecase_certification.py --use-case 10k-analyst \
  --dataset certification/financebench-sample --queue-failures
```

Two details in these gates are worth pausing on.

**Tool trajectory is a certification dimension.** The `tool_use_correctness` evaluator uses the dataset's `question_reasoning` metadata to check that numerical questions were actually answered through the calculator tool. An agent that guesses the right ratio without computing it gets the answer marked correct and the trajectory marked wrong, and the trajectory is what fails it. A system that happens to be right does not generalize; a system that is right for the right reasons does.

**Compliance is a gate, not a metric.** For `advisory-draft`, `regulatory_compliance` is set to 100%: a single prohibited phrase fails the entire certification. To prove the gate does its job, the repository ships a purpose-built adversarial dataset: ten client-update briefs, three compliant controls plus seven that each tempt a distinct failure mode a compliance officer would recognize (guaranteed profits, explicit buy and sell calls, no-risk reassurance, certainty about forward-looking statements, portfolio concentration advice, even a hint of material non-public information). Sets like this are not benchmarks. They start small and curated, and they grow with every reviewed production failure through the feedback loop below.

The verified runs tell the story from both sides. With the temptation flag on, seven of the ten drafts slipped into prohibited phrasing: completeness 100%, tool use 100%, groundedness above its bar, and the certification still FAILED, on compliance alone. Run clean, the same agent resisted every temptation and PASSED with 100% compliance. That is the point. A grounded, complete, well-written draft is still uncertifiable, and your pipeline should be able to demonstrate that to an auditor on demand.

## A portal for the people who sign off [#certification-portal]

The people who approve models do not read traces. Risk officers and business owners want one screen that answers three questions: what is certified, against which dataset, and when was it last checked. So the repository ships a Certification Portal, a small React dashboard on a FastAPI backend that renders exactly that: the latest certification status for every model, variant, and agent, with per-run evaluator breakdowns, run history, and per-item drill-downs behind each row.

The important part is what the portal does not have: a database. Every number on that screen is fetched live from Langfuse's [public REST API](/docs/api-and-data-platform/features/public-api) (datasets, runs, traces, scores), and every row deep-links back to the underlying traces in Langfuse as the source of truth.

Every row on the matrix drills down. Click into a run and you get the run-level breakdown: each gate dimension against its threshold, plus min, max, and pass rate per evaluator:

One more click and you are at per-item granularity: every dataset item scored across all dimensions, each with a trace link that lands on the exact nested trace in Langfuse. This works because Langfuse is API-first by design. The docs put it plainly: "All Langfuse data and features are available via the API." The portal uses that [public API](/docs/api-and-data-platform/features/public-api) through the typed [Python SDK](/docs/api-and-data-platform/features/query-via-sdk), while the [Metrics API](/docs/metrics/features/metrics-api) and [scheduled exports](/docs/api-and-data-platform/features/export-to-blob-storage) support aggregate and warehouse workflows. Langfuse stays the source of truth; the portal is a domain-specific view for the people making the approval decision. Connecting the same evidence to a GRC system still requires authentication, data mapping, access controls, and the institution's review process.

## Keeping the evidence current [#keeping-the-evidence-current]

Certification in financial services is not one-and-done, so the pipeline closes the loop in both directions.

**Production feeds the golden datasets.** A monitoring script scores live traces for compliance and completeness and routes violations to a Langfuse [annotation queue](/docs/evaluation/evaluation-methods/annotation-queues) for human review. The queue itself encodes the routing policy, and reviewers score each item against dedicated human score configs:

<Frame fullWidth>
  ![The Certification Review annotation queue in Langfuse, with pending and completed human reviews](/images/blog/2026-07-15-llm-certification-financial-services/langfuse-annotation-queue.png)
</Frame>

A reviewed failure can then be promoted into the golden dataset as a new item (the reviewer supplies the correct answer; the pipeline never copies the suspect output as ground truth), so the next certification run regression-tests exactly the failure production found.

**Shipping a prompt re-certifies it.** The pipeline ships a GitHub Actions workflow for this loop. Once the one-time Langfuse GitHub Repository Dispatch integration and repository secrets are configured, promoting a prompt to `production` triggers the affected certification with a strict exit code. A prompt promotion that regresses the gate fails the workflow, creating a visible release block for the accountable reviewer instead of quietly shipping to clients.

## What the governance frameworks actually require [#what-regulators-actually-expect]

This pipeline is engineering infrastructure, not regulatory approval. Applicability depends on the institution, jurisdiction, model classification, and use case. The useful question is narrower: which parts of a governance process can this pipeline support with repeatable evidence?

| Framework | Scope and expectation | Evidence this pipeline can support |
| --- | --- | --- |
| [Fed SR 26-2](https://www.federalreserve.gov/supervisionreg/srletters/SR2602.htm) / [OCC 2026-13](https://www.occ.gov/news-issuances/bulletins/2026/bulletin-2026-13.html) | The revised US model risk guidance covers traditional quantitative models and non-generative, non-agentic AI. It explicitly excludes generative and agentic AI from scope, while telling banks to determine appropriate governance and controls for systems outside the guidance | The validation, documentation, monitoring, and effective-challenge patterns are useful design references for an internal LLM governance process. They are not direct compliance requirements for the agents in this post |
| [PRA SS1/23](https://www.bankofengland.co.uk/prudential-regulation/publication/2023/may/model-risk-management-principles-for-banks-ss) | Applies to in-scope UK firms with internal model approval and includes AI and ML modelling techniques where the broader model-risk principles apply: identification, tiering, governance, validation, and mitigants | Versioned certification runs can contribute validation and revalidation evidence, subject to the firm's model inventory, materiality assessment, and independent review |
| [EU AI Act](https://eur-lex.europa.eu/legal-content/EN/TXT/?uri=CELEX:32024R1689) | Classifies specific uses as high risk, including systems used to evaluate the creditworthiness of natural persons. Applicable high-risk systems carry logging, human-oversight, and post-market-monitoring obligations | Traces can contribute logging evidence, annotation queues can support documented human review, and production monitoring can feed a post-market process. None of those controls satisfies the relevant obligation by itself |
| [DORA](https://eur-lex.europa.eu/eli/reg/2022/2554/oj) | Requires in-scope EU financial entities to manage ICT risk and third-party ICT dependencies. Whether and how a model API falls into that program depends on the service and the institution's use of it | Tracing and monitoring can provide operational evidence about availability, degradation, incidents, and third-party model behavior |
| [NIST AI RMF](https://www.nist.gov/itl/ai-risk-management-framework) | A voluntary framework organized around Govern, Map, Measure, and Manage, including documented testing before deployment and monitoring while systems are in use | Datasets, evaluators, gates, review queues, and run history provide technical evidence for Measure and inputs to Manage |

Across the applicable rules and voluntary guidance, the recurring engineering requirement is evidence: versioned artifacts, complete logs, documented human review, and monitoring that continues after deployment. Langfuse can hold that evidence and make it reproducible. The institution still owns applicability decisions, policy, independent challenge, approvals, and legal compliance.

## Adapting for your use case [#adapting-for-your-use-case]

- **Custom datasets**: replace FinanceBench with your institution's own golden Q&A pairs. The setup script accepts any JSON file with input/expected_output/metadata fields.
- **Domain-specific evaluators**: add evaluators that check for your institution's terminology, formatting requirements, or regulatory constraints.
- **Domain-adapted variants**: certify the same model with a specialized system prompt and compare it against the baseline. In our runs, a finance-expert system prompt lifted Claude Sonnet from 90% to 100% numerical accuracy on the FinanceBench sample, and the two variants show up side by side as separate certification rows.
- **Your own agents**: register a new agent with its steps and gate thresholds, and it gets the same nested tracing and multi-dimensional certification as the built-in three.
- **CI/CD integration**: a pytest gate and the GitHub Actions workflows fail your build if certification does not pass, so model and prompt approvals become part of your deployment pipeline.

The full code, including sample data for offline testing and the CI workflows, is in the [langfuse-llm-certification-finance](https://github.com/doneyli/langfuse-llm-certification-finance) repository. If your model approvals still take two weeks of manual testing, the backlog is not the real problem. The real problem is that by the time the approval lands, the system it approved no longer exists.

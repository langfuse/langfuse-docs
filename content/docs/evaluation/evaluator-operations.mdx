# Evaluator Operations

This page is an operations guide for running [LLM-as-a-Judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) and [code evaluators](/docs/evaluation/evaluation-methods/code-evaluators) in production. It covers how evaluator executions work, a troubleshooting checklist for the most common issue (an active evaluator that produces no scores), targeting and scoping, backfilling historical data, judge model connections, limits, and how to alert on evaluation scores with monitors.

If you are setting up an evaluator for the first time, start with the [LLM-as-a-Judge setup guide](/docs/evaluation/evaluation-methods/llm-as-a-judge) instead.

## How evaluator executions work [#how-evaluator-executions-work]

An evaluator execution passes through five stages, and each stage is a place where scores can silently stop appearing. Understanding the pipeline makes the [checklist below](#no-scores-checklist) faster to apply.

1. **Trigger.** At ingest time, each incoming observation or trace is checked against the evaluator's filters. Matching items are added to an evaluation queue. Items that do not match are skipped without an error.
2. **Sampling.** The configured sampling percentage decides whether a queued match is actually evaluated. A 5% sample rate means 95% of matching items never produce a score, by design.
3. **Variable mapping.** Langfuse extracts the data for each `{{variable}}` in the evaluator prompt from the matched item, optionally through a JSONPath expression. All mapped variables are required; if a mapped field is missing on the item, the execution errors instead of producing a score.
4. **Model call.** For LLM-as-a-Judge, Langfuse calls the judge model through the configured [LLM connection](/docs/administration/llm-connection). This call is made server-side by Langfuse, not by your application.
5. **Score write.** The resulting score is attached to the target: the matched observation for observation-level evaluators, the trace for trace-level evaluators, or the experiment item for experiment evaluators.

Every execution is visible in the evaluator's log, and each LLM-as-a-Judge execution creates a full trace in the internal environment `langfuse-llm-as-a-judge` (code evaluators use `langfuse-code-eval`). Executions carry one of four statuses:

| Status    | Meaning                                                                                       |
| --------- | --------------------------------------------------------------------------------------------- |
| Completed | The evaluation finished successfully and wrote a score.                                       |
| Error     | The evaluation failed. Click the execution trace ID for details.                             |
| Delayed   | The evaluation hit LLM provider rate limits and is being retried with exponential backoff.   |
| Pending   | The evaluation is queued and waiting to run.                                                  |

## Checklist: evaluator is active but produces no scores [#no-scores-checklist]

Work through these checks in order. The first branch depends on whether the evaluator log contains entries.

1. **Open the evaluator's log.** Go to the Evaluators page, open your evaluator, and check the Logs tab. An empty log means nothing matched your configuration (continue with checks 2 to 5). Error entries mean matching works but executions fail (jump to checks 6 and 7). Pending or Delayed entries mean scores are coming but slowly (see check 8).
2. **Verify your SDK version or ingestion method.** Observation-level evaluators only run on data ingested via the [OTEL endpoint](/integrations/native/opentelemetry). You need Python SDK v3+ or JS/TS SDK v4+, or direct OTEL ingestion. Data sent through the legacy ingestion API or legacy SDKs (Python v2, JS/TS v3) is never matched. Check your version with `pip show langfuse` or `npm list langfuse`, and see the [Python v2 to v3](/docs/observability/sdk/upgrade-path/python-v2-to-v3) and [JS/TS v3 to v4](/docs/observability/sdk/upgrade-path/js-v3-to-v4) migration guides if you need to upgrade. This is the usual cause when a manual test run works but the evaluator never fires on live traffic.
3. **Check attribute propagation if you filter by trace attributes.** Observation-level evaluators check filters like `userId`, `sessionId`, `tags`, and `metadata` on the observation itself; they do not look up the parent trace. Use `propagate_attributes()` (Python) or `propagateAttributes()` (JS/TS) so these attributes exist on the observations. See [add attributes to observations](/docs/observability/sdk/instrumentation#add-attributes) and the detailed [FAQ on evaluators that do not execute](/faq/all/observation-eval-not-executing) for code examples.
4. **Compare your filters against real data.** Names, observation types, tags, and metadata are matched exactly. Open a trace in the UI, click the observation you expect to be evaluated, and compare its name and type against your filter. The preview in the evaluator setup shows matching data from the last 24 hours; an empty preview on live traffic means the filter is wrong, but a populated preview with an empty log points back to check 2 or 3.
5. **Check your sampling rate against your traffic.** A low sampling percentage combined with low traffic looks like a broken evaluator. At 5% sampling, 100 matching observations produce roughly 5 scores. Raise the sample rate temporarily to confirm the pipeline works.
6. **Check the variable mapping for missing fields.** If the log shows errors, click into an entry. A mapped field that does not exist on the matched item (for example, a metadata key that only some observations carry) fails the execution. Tighten filters so only items with all mapped fields match, or map variables to fields that are always present, such as the observation `input` and `output`.
7. **Verify the LLM connection.** In Project Settings, under LLM Connections, confirm the API key is valid and not expired, and that the model supports structured output. Langfuse parses judge results from structured output, so a model or gateway without it fails every execution. Details are in the [model connections section](#model-connections) below.
8. **Check execution status for delays.** Pending means queued; Delayed means the LLM provider is rate-limiting and Langfuse retries with exponential backoff. Long-running backlogs are most common with legacy trace-level evaluators, which take minutes per evaluation instead of seconds. If delays persist, consider [migrating to observation-level evaluators](/faq/all/llm-as-a-judge-migration) and raising your provider's rate limits for the judge model.

If you complete the checklist and still see no scores, reach out via the [support channels](/support) with your evaluator ID and a sample trace ID.

## Targeting and scoping [#targeting-and-scoping]

Evaluators run on one of three targets, and the target determines both what data the judge sees and where the score is written.

| Target                  | Use for                                            | Score is attached to        |
| ----------------------- | -------------------------------------------------- | --------------------------- |
| Observations (recommended) | Live production monitoring of individual operations | The matched observation     |
| Traces (legacy)         | Legacy SDK setups (Python v2, JS/TS v3)            | The trace                   |
| Experiments             | Offline tests against datasets                     | The traces generated by the experiment run |

Scoping controls which items within a target are evaluated:

- **Filters** narrow the matched set. Observation-level evaluators can stack observation filters (type, name, metadata) with trace-level filters (`userId`, `sessionId`, `tags`, `version`, trace name), provided the trace attributes are [propagated to observations](/faq/all/observation-eval-not-executing).
- **Environments** can be filtered on, for example to exclude non-production environments from evaluation. Evaluator executions themselves are recorded in internal environments (`langfuse-llm-as-a-judge`, `langfuse-code-eval`) that are hidden from the default tracing view.
- **Sampling** runs the evaluator on a percentage of matching items to control cost and throughput.
- **Context boundaries** matter for observation-level evaluators: they only see fields on the matched observation, never sibling or child observations. If the judge needs the end-to-end request and response, target a root observation that records them. See [context available to observation-level evaluators](/docs/evaluation/evaluation-methods/llm-as-a-judge#observation-evaluator-context).
- **Sessions** cannot be targeted directly, because Langfuse does not know when a session has concluded. The common pattern is to tag the final trace of a conversation (for example `conversation_end`) and scope the evaluator to that tag. See [how to evaluate sessions](/faq/all/evaluating-sessions-conversations).
- **Dataset runs** have their own scoping: the evaluator is configured with a source dataset, a scope (new runs, past runs, or both), and a sampling rate. See [evaluators on dataset runs](/faq/all/langfuse-evaluators-on-dataset-runs).

Evaluators and their targeting rules can also be managed programmatically. The API splits the setup into versioned evaluators (how to score) and evaluation rules (what to evaluate); see the [programmatic setup section](/docs/evaluation/evaluation-methods/llm-as-a-judge#api).

## Running evaluators on historical data (backfill) [#backfill]

A newly created or newly updated evaluator only processes data ingested after it becomes active, unless you explicitly backfill. Langfuse supports three backfill flows:

- **Observation-level evaluators** are backfilled from the Traces table. Enable the [Fast Mode](/docs/v4) toggle for the evaluator, filter the Traces table to the timeframe and criteria the evaluator targets, select the matching rows, and click Actions, then Evaluate. Scores are attached to the matching observations inside each selected trace. The step-by-step flow is documented in the [LLM-as-a-Judge guide](/docs/evaluation/evaluation-methods/llm-as-a-judge).
- **Trace-level evaluators (legacy)** offer a choice at setup time between running on new traces only or including existing traces.
- **Dataset-run evaluators** can be scoped to execute on past dataset runs in addition to new ones.

Evaluators managed [via the API](/docs/evaluation/evaluation-methods/llm-as-a-judge#api) are versioned: creating an evaluator under an existing name produces the next version, and active evaluation rules automatically move to it. New versions apply to newly evaluated data; to rescore historical data with an updated version, run the backfill flow above.

## Model connections for the judge [#model-connections]

LLM-as-a-Judge needs an [LLM connection](/docs/administration/llm-connection) that Langfuse can call server-side. Three requirements cause most connection-related failures:

1. **Structured output support.** The judge model must support structured output so Langfuse can parse the score and reasoning. This is required for the default model configured for evaluations.
2. **Tool calling for gateways.** If you connect through an OpenAI-compatible gateway such as LiteLLM, OpenRouter, or Portkey, the gateway must support tool calling in the OpenAI format. The [LLM connections page](/docs/administration/llm-connection) includes the exact request shape your gateway must handle.
3. **Network reachability from Langfuse.** The judge call originates from the Langfuse deployment, not from your application. On self-hosted deployments, the LLM API or gateway must be reachable from the Langfuse containers, for example deployed in the same VPC or a peered VPC. On Langfuse Cloud, the model endpoint must be reachable from Langfuse Cloud over the public internet. If your provider endpoint only accepts traffic from your private network (for example, an Azure OpenAI resource with public network access disabled), Langfuse Cloud cannot reach it and every judge call fails, even though the same credentials work from inside your network.

Supported providers are OpenAI, Azure OpenAI, Anthropic, Google AI Studio, Google Vertex AI, and Amazon Bedrock, plus any provider or proxy that implements the OpenAI API schema. Amazon Bedrock requires the `bedrock:InvokeModel` and `bedrock:InvokeModelWithResponseStream` permissions.

Model parameters such as `temperature`, `top_p`, and `max_tokens`, and provider-specific options such as `reasoning_effort`, are not set on the connection in Project Settings. They are configured when you select the LLM connection during evaluator setup (or in the Playground). This applies to managed evaluators from the catalog as well: pick the evaluator, then adjust the model parameters in the model settings of the evaluator configuration. A custom evaluator can additionally pin a dedicated model that overrides the project's default evaluation model.

## Limits and quotas [#limits]

The documented limits relevant to evaluator operations, as of July 2026:

| Limit                                | Value                                                                                      |
| ------------------------------------ | ------------------------------------------------------------------------------------------ |
| Monitors per organization (Cloud)    | 2 (Hobby), 20 (Core), 50 (Pro), 100 (Enterprise)                                           |
| API payload size                     | 5 MB per request and 5 MB per response                                                     |
| API rate limits                      | Applied per organization and plan, see [API limits](/faq/all/api-limits)                   |
| Code evaluator runtime               | 2 seconds per execution, no network egress, standard library only                          |
| Code evaluator sizes                 | 256 KB source, 5.5 MB dispatch payload, 256 KB result                                      |
| Automation delivery failures         | After 5 consecutive failures, the automation's trigger is disabled until re-enabled        |

The full code evaluator constraints are listed under [runtime constraints](/docs/evaluation/evaluation-methods/code-evaluators#runtime-constraints). Self-hosted instances have no hard platform limits; capacity depends on your infrastructure.

If you run into unexpected limits, for example when managing many active evaluators through the API, or if you need a limit increased, reach out via the chat widget or [support](/support).

## Monitors and alerts on evaluation scores [#monitors-on-scores]

Once evaluators write scores, [monitors](/docs/metrics/features/monitors) turn them into alerts. Monitors support numeric, categorical, and boolean scores as data sources; boolean score support was added in July 2026. For boolean scores, the average value is the share of scores that are `true`, so "alert when the hallucination rate exceeds 5%" is a monitor on the average of a boolean score with the `>` operator and a `0.05` threshold.

The threshold semantics are worth spelling out, because they are the most common source of confusion:

- The **operator** (`>`, `≥`, `<`, `≤`, `=`, `≠`) defines the direction of the alert condition, and crossing the required **alert threshold** sets severity to ALERT.
- The optional **warning threshold** sits before the alert threshold and sets severity to WARNING.
- The **window** defines how far back each evaluation looks (for example 1 hour or 1 day); the monitor evaluates the aggregated metric over that window, not individual scores.
- Notifications fire on severity transitions: entering WARNING or ALERT always notifies, and recovering to OK always notifies. A monitor that stays in ALERT does not re-notify unless **Renotify** is enabled (every 1 to 10,080 minutes).
- When the query returns **no data**, the default mode treats the missing value as 0 and compares it against the thresholds. For a monitor like "average correctness < 0.7", a quiet period therefore triggers an alert. Choose one of the other no-data modes (keep previous severity, show NO_DATA, or notify after sustained NO_DATA) if that is not what you want.

Alerts are routed through automations to Slack, webhooks, or GitHub Actions. See the [monitor setup guide](/docs/metrics/features/monitors#monitor-setup) and [alert states](/docs/metrics/features/monitors#alert-states) for the full reference.

## FAQ [#faq]

<Details>
<Summary>Why does my evaluator work in the setup preview but not on live traffic?</Summary>

The preview and live execution use different paths. The preview reads historical data from the last 24 hours, while live matching happens at ingest time on the incoming data. If the preview shows matches but no scores appear, the usual causes are an incompatible ingestion path (legacy SDKs or the legacy ingestion API are not matched by observation-level evaluators) or trace attributes that are not propagated to observations. Work through the [checklist above](#no-scores-checklist) in order.

</Details>

<Details>
<Summary>Why are my evaluation scores delayed?</Summary>

Check the execution status in the evaluator log. Delayed means the LLM provider is rate-limiting the judge model and Langfuse retries with exponential backoff; Pending means the execution is queued. Observation-level evaluators complete in seconds, while legacy trace-level evaluators take minutes per evaluation and can build up backlogs. If you see persistent delays on trace-level evaluators, [upgrade them to observation-level](/faq/all/llm-as-a-judge-migration).

</Details>

<Details>
<Summary>Can I run an evaluator on traces ingested before the evaluator existed?</Summary>

Yes. For observation-level evaluators, enable Fast Mode for the evaluator, then filter and select the relevant rows in the Traces table and use Actions, then Evaluate. Trace-level evaluators can include existing traces at setup time, and dataset-run evaluators can be scoped to past runs. See [running evaluators on historical data](#backfill).

</Details>

<Details>
<Summary>Can I configure the judge model's temperature?</Summary>

Yes. Model parameters, including `temperature`, `top_p`, and `max_tokens`, are set when you select the LLM connection during evaluator setup, not on the connection itself in Project Settings. This also works for managed evaluators from the catalog. Provider-specific options, such as `reasoning_effort`, can be passed as additional provider options in the same model settings.

</Details>

<Details>
<Summary>Is there a limit on the number of evaluators per project?</Summary>

There is no documented fixed limit on evaluators per project. General [API rate limits](/faq/all/api-limits) apply when managing evaluators programmatically. If you operate a large number of active evaluators and hit unexpected limits, reach out via the chat widget or [support](/support); limits on Langfuse Cloud can be reviewed per plan.

</Details>

<Details>
<Summary>Why did my monitor fire an alert during a period with no traffic?</Summary>

By default, monitors treat missing data as 0. If your alert condition is a lower bound, such as "average score < 0.7", a window with no scores evaluates to 0 and crosses the threshold. Switch the monitor's no-data handling to keep the previous severity or to a NO_DATA mode if quiet periods should not alert. See [monitors and alerts](/docs/metrics/features/monitors#alert-states).

</Details>

<Details>
<Summary>How do I evaluate multi-turn conversations?</Summary>

Evaluators cannot target sessions directly. Ensure the complete conversation history is present on the trace or observation you evaluate (it usually is when the history is passed as LLM input), then either evaluate every trace or tag the final trace of a conversation and scope the evaluator to that tag. Use JSONPath in the variable mapping to select the relevant part of the conversation payload. See [how to evaluate sessions and conversations](/faq/all/evaluating-sessions-conversations).

</Details>

# Interpreting Experiment Results [#interpreting-experiment-results]

<Callout type="info">
  **Beta:** Experiment analysis in the UI is currently in beta. Features and workflows are evolving based on feedback.
</Callout>

When developing AI agents, every change to your application—whether a prompt adjustment, a model swap, or a tool update—produces a new experiment. Comparing experiments helps you determine if changes represent genuine progress or introduce regressions.

The recommended approach is to designate a **Baseline** (your current reference, often production) and compare **Candidates**—the experiments produced by your changes—against it. This makes performance deltas immediately visible and ensures you catch regressions before they reach users. You can also compare experiments side-by-side without a baseline for exploratory analysis.

<Callout type="info">
  **Recommended:** To compare experiments meaningfully, run [evaluators](/docs/evaluation/overview) on your experiments to generate scores. See [Evaluation Methods](/docs/evaluation/evaluation-methods) to learn how to set up evaluators for experiments.
</Callout>

## Access the Comparison View [#access-comparison]

1. Click **Experiments** in the navigation bar to see all experiments in your project.
2. Use filters to narrow down by dataset, metadata, or configuration.
3. Click on your intended **Baseline** experiment to open it.
4. In the right sidebar, select the experiments to compare against—or use multi-select from the list and click **Compare**.

### Regression-First Analysis [#regression-first]

Once in comparison mode, all metrics display relative to your Baseline. Prioritize checking for regressions first—in agentic workflows, improvements in one area often cause failures elsewhere. Filter by score thresholds to surface problem areas, then validate that aggregate scores show improvement on core metrics.

### Multi-Candidate & Spreadsheet-Like Comparison [#multi-candidate]

When testing multiple hypotheses (e.g., `Prompt_A`, `Prompt_B`, `Prompt_C`), you can view runs side-by-side without a fixed baseline to observe performance trends across the cohort. This "spreadsheet-like" view is useful if you have not yet established a baseline, or simply want to compare results at a glance. You may **clear the baseline at any time** to enable this side-by-side comparison mode.

> **Tip:** Clearing the baseline transforms the UI into a flexible table, making it easy to scan, filter, and compare multiple experiment results just like in a spreadsheet—even before a definitive baseline is chosen.

## Regression Triage [#regression-triage]

When a candidate fails to outperform the baseline, use these filters for post-mortem analysis:

| Filter | Purpose |
|--------|---------|
| **Score Thresholds** | Filter runs below critical quality bars (e.g., `Accuracy < 0.7`) |
| **Metadata Cohorts** | Identify if failures localize to specific input types (e.g., `context: long`) |
| **Trace Inspection** | Move from aggregate metrics to individual traces for qualitative analysis |

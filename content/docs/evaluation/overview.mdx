# Evaluation Overview

Evals give you a repeatable check of your LLM application's behavior. You replace guesswork with data.

LLM evaluation is part of the [AI engineering loop](/academy/ai-engineering-loop):

1. **Trace** -> Capture real behavior with [Observability](/docs/observability/get-started).
2. **Monitor** -> Track production quality with online evaluators and [Score Analytics](/docs/evaluation/scores/score-analytics).
3. **Build datasets** -> Turn traces into reusable evaluation assets with [annotations](/docs/evaluation/evaluation-methods/annotation-queues) and [datasets](/docs/evaluation/experiments/datasets).
4. **Experiment** -> Validate changes with [experiments](/docs/evaluation/experiments/experiments-via-sdk) and [CI/CD checks](/docs/evaluation/experiments/experiments-ci-cd) before they ship.
5. **Evaluate** -> Decide what is good enough to ship with [evaluation methods](/academy/evaluate).

## Trace

Capture what users asked, what your system did, and where the output came from with [traces and observations](/docs/observability/get-started). Trace data enables the rest of the loop: monitoring, datasets, and regression checks. For the broader pattern, see [tracing](/academy/tracing).

## Monitor

Use production signals to find the traces worth reviewing. [Monitoring](/academy/monitoring) helps you spot data drift, flag quality issues automatically, and discover examples for your evaluation set.

- Track trends, data drift, and regressions in [Score Analytics](/docs/evaluation/scores/score-analytics) and [custom dashboards](/docs/metrics/features/custom-dashboards).
- Score live behavior with [LLM-as-a-Judge](/docs/evaluation/evaluation-methods/llm-as-a-judge), custom [scores via API/SDK](/docs/evaluation/evaluation-methods/scores-via-sdk), or user feedback.

## Build datasets

Turn raw traces into reusable evaluation assets. Start with human review, name the failure modes, then convert useful examples into datasets and score definitions. Start with [datasets](/academy/datasets) and [error analysis](/academy/monitoring/error-analysis).

- Use [Annotation Queues](/docs/evaluation/evaluation-methods/annotation-queues), [Scores via UI](/docs/evaluation/evaluation-methods/scores-via-ui), and [`TEXT` scores](/docs/evaluation/scores/overview#score-types) for open coding.
- Group notes into failure modes (axial coding), then turn stable categories into [structured labels](/docs/evaluation/scores/overview#score-types) and [evaluation criteria](/faq/all/manage-score-configs).

## Experiment

Use [experiments](/academy/experiments) to confirm that a prompt, model, retrieval setup, agent implementation, or evaluator variant improves quality without regressions.

- Compare candidates on the same [dataset](/docs/evaluation/experiments/datasets) and scoring criteria with [experiments via UI](/docs/evaluation/experiments/experiments-via-ui) for prompt and model changes, or [experiments via SDK](/docs/evaluation/experiments/experiments-via-sdk) for application logic.
- Run regression checks manually or in [CI/CD](/docs/evaluation/experiments/experiments-ci-cd) before you ship to production.

## Evaluate

Judge experiment outputs before shipping. Start with manual review to understand quality and failure modes, then automate dedicated evaluators where they add repeatable signal. The tradeoffs are covered in [evaluation methods](/academy/evaluate).

- Use [manual evaluation](/docs/evaluation/evaluation-methods/scores-via-ui) to build intuition and calibrate automated evaluators.
- Use [scores via API/SDK](/docs/evaluation/evaluation-methods/scores-via-sdk) for custom evaluation pipelines, guardrails, runtime checks, user feedback, and internal review workflows.
- Use [LLM-as-a-Judge](/docs/evaluation/evaluation-methods/llm-as-a-judge) for qualities that require language understanding, such as relevance, tone, completeness, or factuality.
